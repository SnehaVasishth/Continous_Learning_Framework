Web app gap analysis vs RFP deliverables

A structured comparison between what The Hackett Group has committed to Keysight in the RFP response package (Approach deck, Executive Summary, Solution Overview, Implementation Approach, AI/ML Capabilities) and what the ZBrain demo application currently delivers. The analysis identifies seven categories of gap, sequences a four-week plan to close them, and flags the four explicit asks called out for immediate execution.

The Hackett Group on ZBrain · Internal · Response date 2026-05-13

Summary at a glance

The web app delivers a credible six-stage pipeline, four-gate confidence tiering, HITL queue, audit trace, drift alerts, A/B experiments, and the Cost and Benefit Calculator. The largest gaps versus the RFP package are:

  1. Application-level governance — the deck commits to eleven AGT enforcement layers and OWASP ASI 2026 10/10 coverage; the app currently exposes around three.
  2. Confidence model formula mismatch — deliverables say composite = min(gates) across four gates; code uses a weighted average across three components.
  3. Scale credibility — the proposal commits to 2,000 emails per day and a 50x stress test; the demo holds 138 emails total and runs synchronously.
  4. Seed depth — Salesforce data, mailboxes, languages, and regions are sparse compared to the operational footprint the proposal positions.
Sequenced plan. Four phases over four weeks if parallelised, six to seven weeks if single-threaded. Phase 1 (foundation seed expansion, realistic email content, long threads) unlocks everything downstream and runs first.

The deliverables promise eleven named AGT enforcement layers with full OWASP ASI 2026 coverage (10/10) and Agent Context Protocol exposing role, duty, data touch points, metrics, policy, and retention for every agent. The current app surfaces almost none of this.

Category B · Confidence model formula mismatch CRITICAL

Defensibility risk: deliverables and code use different math.
  • Deliverables say: composite confidence = min(gates), four gates: Classification, Extraction, Entity Resolution, Action Feasibility.
  • Code says (CLAUDE.md and decide.py): composite = 0.45 × intent + 0.35 × extract + 0.20 × customer-match, three components, no Action Feasibility gate.

A sharp procurement reviewer will catch this. Fix the code to use min(gates) across all four gates named in the deliverables. The Action Feasibility gate (can the downstream write actually execute with what was resolved) needs to be a real check against Salesforce and Oracle write-path field requirements.

Category G · Four explicit asks PRIORITY

Items called out for immediate execution.

G1 · Longer email threads (10 to 40 messages each)

Current state. Most seed emails are single messages or short 2 to 3 message threads.

Action.

  • Extend the seed corpus to include at least 20 long threads ranging from 10 to 40 messages each. Mix of: multi-turn purchase order clarifications, service work order updates with several CSR responses, escalation chains that bounce between regions, and quote-to-order conversations spanning weeks.
  • Update the Inbox UI to render thread depth as a clear count and a collapsible message list.
  • Update the Activity and Trace views to show thread context: every prior message that informed the classification, with the system's reading of each message.
  • Add a Thread Length filter to the Inbox and a Dashboard tile for average thread length and longest open thread.

G2 · Comprehensive email content

Current state. Seed emails are short, often a sentence or two. Real customer mail is multi-paragraph with greetings, context, inline attachment references, and signature blocks.

Action.

  • Rewrite seed email bodies to be realistic in length and structure: greeting, context paragraph, request paragraphs, attachment reference, sign-off, signature block, occasional footer disclaimer.
  • Vary tone: formal procurement, hurried operations, technical support escalation, polite follow-up, regulatory compliance request.
  • Include realistic headers (caution banners, forwarded headers, reply chains, CC lists).
  • Embed inline HTML tables for line-item POs to test mixed-mode extraction.
  • Include emails in EN, ES, JA, and add at least 5 more languages (French Canadian, German, Italian, Mandarin, Japanese hankaku and zenkaku variants) to back the 16 to 18 language commitment.

G3 · Parallel processing for 2,000-per-day scale

Current state. Pipeline runs are synchronous; no demonstration that the system handles concurrent inbound at scale.

Action.

  • Worker pool. Introduce a Celery, RQ, or asyncio worker pool fed by a job queue. Each inbound email becomes a job; N workers process in parallel.
  • Throughput dashboard. Add Dashboard tiles for emails-per-minute, queue depth, current worker count, p50/p95/p99 pipeline latency.
  • Stress-test script. A make stress-test target that fires 2,000 emails into the system in 1 hour (about 33 per minute, the 2,000/day baseline) and a 50x burst variant (1,667 per minute) the deliverables commit to.
  • Visible scale demo. A Scale Test button on the Settings page that runs a controlled stress test and reports the result inline.
  • Backend. Switch SQLite to WAL mode at minimum; consider Postgres for the demo if you want to defend the 100-emails-per-second claim in the implementation risk register.

G4 · Salesforce data seed expansion

Current state. 10 customers, 23 products, 11 orders. Sparse against a 650-FTE operation handling 880k emails per year.

Action.

  • About 200 customers with realistic Account hierarchies (parent and child, billing vs shipping, regional ownership).
  • About 500 contacts linked to customers with realistic name and email patterns including aliases.
  • About 100 products covering the full Keysight catalog spectrum (PNA-X, MXG, FieldFox, Infiniium UXR, BERT, SMU, plus service SKUs, calibration certs, training credits).
  • About 300 quotes across active, expired, won, lost.
  • About 500 orders across booked, in-fulfilment, shipped, invoiced, on-hold so the Existing-CCC matrix can exercise all 10 statuses.
  • About 150 work orders covering calibration, repair, installation.
  • About 50 service contracts active and expiring.
  • About 30 distributors to exercise the distributor list routing rule.
  • Magic-SKU records for CUSTOM PRODUCT, SOWDUMMY, EXPORTDUMMY.
  • Re-seed so emails reference real Salesforce IDs across the expanded catalog.
  • Ensure all 4 regions (Americas, EMEA, Asia-Pac, Japan) have customers, contacts, products, and orders. Add Japan-specific records with hankaku and zenkaku variants.
