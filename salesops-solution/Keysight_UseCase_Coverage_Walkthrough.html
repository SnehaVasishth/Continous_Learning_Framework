How to read this document

Each use case has the same three pieces:

  1. What it is in plain language — what the customer wants when this kind of email arrives.
  2. How the system handles it — step-by-step, no jargon.
  3. Live examples — real pipelines we ran during the validation batch. Each one links to its trace.

You'll see three statuses on each example:

  • Auto-completed the AI handled the whole thing — no human action needed.
  • Awaiting CSR the AI did the work it could and parked the case in the right human queue for a CSR to review or approve.
  • AIOA fallout the AI Order Acceptance app (the external Keysight system that validates POs) flagged a problem and routed the case to its own review queue.

For each case, the example links go to https://app.solution.zbrain.ai/keysight-salesops/trace/<number> — that's the live trace page showing every step the AI took, what it extracted, what it decided, and which CSR queue (if any) it routed to.

Use Case 1 — Trade Order Entry

What it is

A customer sends an email with a Purchase Order — they want to buy something. The PO might be a fresh order, an export-restricted item, or a conversion of an earlier quote. The AI needs to read the PO, confirm we recognise the customer in Salesforce, validate the order against the AIOA rule book, generate a Sales Order Acknowledgment, and either auto-process or route to the Trade desk.

How the system handles it

  1. Reads the email and classifies it as a PO intake.
  2. Extracts PO number, line items, requested ship date, payment terms.
  3. Looks up the customer in Salesforce.
  4. Checks if there's an FCNV review gate to trip (do we have enough information?).
  5. Triggers the external AIOA validation app via webhook. AIOA owns the rest from here.
  6. If AIOA accepts the order: the case is handed off — AIOA proceeds to Oracle EBS booking and sends the customer the SOA. ZBrain does no further customer comms.
  7. If AIOA rejects the order: the case lands in AIOA's own Fallout Review queue. An AI OA specialist works it inside AIOA. Again, ZBrain does no further customer comms.

Use Case 2 — Trade Sales Change Order

What it is

A customer asks to change something on an order they already placed — quantity, line item, price tier. The AI needs to recognise this is a change request (not a new order), open a Change Order CCC Request, route it to the Trade desk, and surface the existing order + the proposed change.

How the system handles it

  1. Classifies the email as a trade change order request.
  2. Extracts the order reference + the proposed change details.
  3. Creates a Salesforce Case shaped as "Change Order received".
  4. Assigns the case to the Trade CSR queue (or FCNV CSR if details are thin).
  5. The CSR confirms the change, updates the Salesforce Order, and notifies the customer.

Use Case 3 — Service Order Management (Work Order Automation)

What it is

A customer requests a service or calibration visit. It may be for one instrument or a whole lab full. The AI needs to recognise this is a service request, identify each asset, create a Salesforce Work Order per asset, attach the customer email to the work order, and close the CCC Request without bothering the customer with another email — the work is now scheduled and the field service team takes over.

How the system handles it

  1. Classifies the email as a service order.
  2. Extracts the asset list (model, serial, location, requested completion date).
  3. For one asset: creates one Salesforce Work Order. For multiple: populates a bulk staging table, then creates one Work Order per asset.
  4. Attaches the customer email + any spreadsheets to each new Work Order.
  5. Closes the CCC Request with no reply. The SOM CSR can audit the auto-created Work Orders separately.

Use Case 4 — Service Order Management — Work Order Update

What it is

A customer wants to add information to a Work Order that's already open — a note for the field engineer, a new attachment, a PO they forgot to include. The AI updates the Work Order directly and closes the case without a customer reply.

How the system handles it

  1. Classifies the email as a Work Order update request.
  2. Identifies the Work Order number from the email.
  3. Updates the existing Work Order (note or task) and attaches the email + any documents.
  4. If the customer attached a PO, the AI also triggers AIOA validation on that PO.
  5. Closes the CCC Request with no reply.

Use Case 5 — Work Order Status Inquiry

What it is

A customer asks "what's the status of my work order?" — they don't want anything done, they want an update. The AI looks up the Work Order, translates the internal status into customer-friendly language, adds a Keysight Standard Procedure reassurance line, and sends the reply automatically.

How the system handles it

  1. Classifies the email as a status inquiry.
  2. Looks up the Work Order in Salesforce.
  3. Translates the internal status (e.g., "in_progress") into a customer-friendly sentence (e.g., "Our field team has begun the calibration and you can expect a status update by <date>.")
  4. Sends the reply automatically. No CCC Request is created on the happy path.

Use Case 6 — Service Contracts

What it is

A customer wants a service contract — preventive maintenance, response-time SLAs, multi-year coverage. The AI routes the case to the Service Contracts (CTA) team, validates any attached PO via AIOA, and lets the specialist handle the contract workflow.

How the system handles it

  1. Classifies the email as a service-contract request.
  2. Identifies the requested term, annual value, coverage scope.
  3. If a PO is attached, triggers AIOA validation; otherwise routes to CTA specialist.
  4. Routes to S&A (Service Contracts & Agreements) specialist or holds in FCNV if details are thin.

Use Case 7 — SSD Change Request

What it is

A customer wants to change the requested ship date (SSD) on an order. The AI adds the change to the Post Order Booking dashboard, notifies the factory and the Sales Order Owner, and lets the factory coordinate the Oracle update.

How the system handles it

  1. Classifies the email as an SSD change request.
  2. Creates a CCC Request and assigns the Sales Order Owner.
  3. Adds the request to the CSR dashboard with factory notification.
  4. Factory coordinates with CSR and triggers the Oracle date change from the dashboard.
  5. Case auto-closes once the factory confirms the Oracle change. No customer email from ZBrain — the factory closes the loop.

How the system catches its own mistakes

Every case the pipeline runs, a built-in self-check looks at the case state at every stage boundary and asks questions like:

  • "If AIOA said no to this order, did we make sure ZBrain didn't also send the customer a reply?"
  • "If we're auto-creating Work Orders for a service request, did we actually attach the email to each one?"
  • "If the case landed in the FCNV queue, did the tier correctly drop to L2 so a human reviews it?"
  • "For an SSD change, did the factory handoff actually fire?"

These questions are written as declarative rules in the Knowledge Base. Operators can edit, version, and roll back rules in the same UI they edit intent definitions and business rules. Before turning a new rule on, they can back-test it against the last 200 pipelines from the rule editor — see how many cases it would have flagged, how many it would have passed, and whether the new rule breaks anything historical.

In this batch run, the system performed 115 invariant checks. 113 passed. 2 were flagged on a single pipeline — a real edge case in the wording of one rule (a Work Order update path where AIOA fires takes a slightly different "completed" status than the rule's predicate expected). That's exactly what the verifier is designed to catch. The fix is a one-line edit in the Knowledge Base — no code change, no deploy.

What the AI is doing under the hood (for engineers): the pipeline runs in six stages — Pre-Intake (Outlook-rule short-circuits), Intake (language detect + intent classify), Extract (OCR + schema-driven extraction + customer match + reconcile), Decide (FCNV review gate + AIOA webhook + four-gate confidence + owner assignment), Execute (workflow action specific to the intent: Salesforce Order/WO/Case writes; AIOA hand-off short-circuit), Communicate (draft + translate + SOA generation + SharePoint filing). Six Salesforce Queues are live in the leewayhertztechnologies org for routing: FCNV, AI OA Fallout, Trade CSR, SOM CSR, CTA Contracts & Agreements, Sales Order Owner / Post Order Booking. Every CSR queue assignment, owner mapping, intent definition, and self-check rule lives in the Knowledge Base — operators edit them in the UI with full version history, restore, and back-test simulation.
