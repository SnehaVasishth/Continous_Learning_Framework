Verifier catches

Only one row (PID 159) recorded blocker violations. The case is wo_update_request with a PO that triggered AIOA validation → AIOA_FAIL. The verifier's wo_update_no_reply rule fired because its invariant requires exec_status == 'applied_no_reply', but the AIOA-fallout path sets exec_status == 'aioa_fallout_queue'. This is exactly the kind of edge case the verifier exists to find — the rule needs widening to accept either outcome. The fix is one KB edit (no code change): change the rule's invariant to no_reply == True AND exec_status in ('applied_no_reply', 'aioa_fallout_queue'). The simulator on the rule editor will let the operator confirm the new predicate passes the existing batch before promoting it from shadow to active.

5. RFP rubrics — also in KB

Each intent in the intent KB namespace carries an rfp_rubric field describing the canonical end-to-end path for that intent. The verifier's optional LLM second-opinion check feeds the relevant rubric to the LLM at runtime. Operators edit the rubric from the KB editor like any other intent field — zero code change. Rubrics ship for: po_intake, quote_to_order, trade_change_order, service_order, wo_update_request, wo_status_inquiry, service_contract_request, ssd_change_request, hold_release, delivery_change, general_inquiry, out_of_scope, spam, kso.

7. What changed in the stages (cumulative)

  • Stage 2 substep 2.6 FCNV Review Gate — fires when required parties are missing; tags case with fcnv_fallout_label = fcnv_scope.
  • Stage 3 substep 3.0c AIOA Webhook — reframed as a webhook handoff (was "validation"). AIOA owns the work; ZBrain just triggers and records the outcome.
  • Stage 3 substep 3.4 Assign CCC Request owner — resolves the owner from owner_mapping KB, computes track + tracks_touched, persists decision.owner with salesforce_owner_id.
  • Stage 4 AIOA short-circuit — when AIOA fires (PASS or FAIL), ZBrain stops at Stage 4. execution.no_reply = True is set so Stage 5 short-circuits.
  • Stage 4 multi-asset SOM auto-WO + UC4 WO update + UC7 SSD factory handoff — explicit no-reply close paths with substep emissions (4.3a / 4.3b / 4.3c / 4.3d).
  • Stage 5 5.0 No-reply close — short-circuits whenever execution.no_reply is set; trace message differentiates AIOA-PASS vs AIOA-FAIL vs SOM/SSD no-reply.
  • Stage 5 5.3 SOA filing to SharePoint — uploads to SalesOps/SOA/, persists web_url in reply.sharepoint_filed, makes the SharePoint link the canonical attachment on the CommunicationLog.
  • Orchestrator — writes Case.OwnerId = salesforce_owner_id on real SF Case writes. Recognises handed_off_to_aioa / aioa_fallout_queue / applied_no_reply as terminal-completed states. Calls the verifier at stage_end and at final close.

8. Connection-monitor + readiness gate

  • Salesforce, SharePoint, and Mailbox health are checked on every poll. Failures publish a critical/warning notification (same kind upserts so polls don't duplicate rows). Resolves automatically the next time the check passes — no manual ack required.
  • Pipeline ingress (REST /run/{id}, /run-batch, retry, and the worker pool submit()) all gate on readiness. Returns HTTP 412 with the full payload when blocked.
  • Mailbox is currently demoted from blocker to warning per the operator's call (synthetic seed data, not live IMAP). Flip MAILBOX_REQUIRED_FOR_READINESS=1 in env to make it strict again.
